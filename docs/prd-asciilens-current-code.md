# AsciiLens 当前产品需求文档

## 问题陈述

创作者、设计师和视觉实验用户想把普通图片快速转成有辨识度的 ASCII 视觉作品，但常见工具通常只提供整图字符化，缺少对主体区域、背景氛围、字符密度、颜色风格和移动端导出的精细控制。用户需要一个打开即可使用的图像到 ASCII 编辑器：既能保留原图主体关系，又能通过局部选区叠加 ASCII、快速调试风格，并最终导出可分享的 PNG。

AsciiLens 当前还承担一个轻量产品学习目标：在不保存图片内容、文件名、IP 地址或原始来源地址的前提下，记录匿名使用事件，让后续智能体或运营流程可以读取汇总指标，判断用户是否真的完成上传、编辑、选区创建和导出。

## 解决方案

AsciiLens 提供一个单页图片 ASCII 艺术工作室。用户进入后直接上传图片，随后在画布预览区和编辑面板中完成编辑。产品支持两种核心渲染模式：`主体叠加` 用于在用户手动圈选的主体或局部区域上叠加 ASCII，`全图 ASCII` 用于将整张图片转换为 ASCII 艺术。用户可以绘制矩形、椭圆或自由形状选区，移动、缩放、旋转、删除选区，并为选区设置独立的字符、颜色、密度、明暗和发光效果。

编辑器保持移动端优先的可用性：画布占据主要视野，控制面板紧凑可滚动，导出在桌面端触发下载，在支持分享文件的移动端优先调用系统分享，否则打开预览让用户保存。产品通过匿名审计事件记录页面访问、图片上传、渲染模式变化、选区创建/删除和 PNG 导出，并通过受令牌保护的汇总接口提供按 `今天`、`昨天`、`近 7 天` 聚合的使用指标。

## 实现决策

- 将 AsciiLens 构建为 Vite + React 单页编辑器，首屏直接进入可用的工作室界面，而不是营销落地页。
- 用户可见的产品命名保持为 `AsciiLens` 或 `ASCII LENS`；不暴露导入模板名或无关工作室名。
- 将编辑器设置建模为单一状态对象，包含渲染模式、全局 ASCII 设置、图片图层设置和规范化选区集合。
- 使用 `主体叠加` 与 `全图 ASCII` 作为两个主要渲染模式。`主体叠加` 保留源图片作为底层，只在用户选区内绘制 ASCII。`全图 ASCII` 在黑色背景上把整张图片渲染为 ASCII。
- 选区以相对于图片的规范化几何信息保存，使画布尺寸和视口变化后选区仍能对齐。
- 支持矩形、椭圆和自由形状选区。自由形状选区保存为闭合点路径。
- 每个已提交选区都有生成的唯一标识，并快照当前 ASCII 样式默认值，从而支持后续独立样式编辑。
- 当编辑器处于叠加模式、存在当前选区且被修改的是选区专属设置时，将样式编辑路由到当前选区；否则更新全局设置。
- 图片图层控件保持为整个构图的全局设置，包括背景滤镜模式、背景颜色、模糊、暗化和暗角。
- ASCII 通过画布降采样和亮度到字符的映射进行渲染。亮度、对比度、明暗反转、字符集、密度、颜色模式、单色值和发光效果都参与渲染流水线。
- 保留一个内部处理画布用于图像采样，并保留一个输出画布用于可见预览和导出。
- 使用输出画布作为导出来源，使导出的 PNG 与可见渲染结果一致。
- 在具备能力的移动浏览器上优先使用原生文件分享，然后降级为直接下载，再降级为打开预览。
- 将分析事件视为非阻塞逻辑。客户端事件发送不得中断图片编辑、渲染或导出。
- 可用时在本地存储中持久化匿名客户端会话标识；不可用时生成临时标识。
- 只发送隐私边界明确的审计载荷：事件名、匿名会话标识、路径、来源主机、语言环境、视口、用户代理和压缩后的元数据。
- 通过服务端接口存储审计数据，而不是从浏览器直接暴露 Supabase 访问能力。
- 当 Supabase 未配置时，审计事件写入返回 `204`，保持应用在未接入分析的环境中仍可正常使用。
- 审计汇总接口使用 Bearer 令牌保护，并且只返回聚合指标。
- 报告范围使用 Asia/Shanghai 日历窗口，支持 `今天`、`昨天` 和 `近 7 天`。
- 通过可视视口更新 CSS 视口变量，并尊重浏览器底部安全区域，使编辑器具备移动端适配能力。
- 在双指舞台手势期间禁用选区编辑手势，避免缩放/平移与选区编辑互相冲突。
- 保持界面控件紧凑稳定：渲染模式在最前；叠加模式下选区工具紧随其后；然后依次是颜色、字符、密度、色调、图片图层和底部操作。
- 维护 AsciiLens 视觉系统：深色舞台、高对比排版、粉色/奶油色/物件绿/电蓝点缀、紧凑圆角控件，并且不显示区域列表或数字选区徽标。
- 通过翻译映射和语言上下文支持英文与中文文案。
- 在导出、重置、新图片、形状工具、缩放/旋转控制点和当前选区删除等场景中优先使用图标化控件。
- 破坏性入口保持紧凑，并严格限定在当前选区或重置操作范围内。

## 测试决策

- 好的测试应验证外部行为：用户能上传什么、看到什么、选择什么、编辑什么、导出什么，以及服务端接口接受或返回什么。测试不应断言私有组件状态、内部钩子顺序或画布循环的具体实现细节，除非这些细节构成确定性渲染契约的一部分。
- 价值最高的编辑器测试接缝是运行中应用的浏览器级交互测试。应覆盖初始上传、预览渲染、渲染模式切换、形状创建、当前选区样式编辑、选区移动、按钮删除、键盘删除、重置效果、新图片替换和导出触发/降级。
- 画布渲染测试接缝应使用确定性图片夹具，检查可观察的画布结果，例如非空输出、输出尺寸，以及切换全图模式和叠加模式后的可见差异。除受控冒烟检查外，应避免脆弱的逐像素快照。
- 响应式视觉测试接缝应在常见桌面和移动端视口下运行浏览器检查，确认没有标签裁切、底部操作隐藏、选区覆盖层错位或舞台尺寸不可用的问题。
- 无障碍测试接缝应验证上传、渲染模式、形状工具、色块、滑杆、导出/重置/新图片操作和当前选区删除都可通过键盘到达，并验证纯图标操作具备可访问名称。
- 审计客户端测试接缝应验证上传、渲染模式变化、选区创建/删除和导出尝试会调用审计接口，载荷经过清洗且隐私边界明确，并且网络失败不会阻塞编辑。
- 审计接口测试接缝应覆盖方法限制、无效 JSON、无效事件名、缺失 Supabase 配置、成功写入时的载荷形状、汇总读取未授权、支持的汇总范围和聚合输出结构。
- 汇总聚合测试接缝应覆盖 Asia/Shanghai 日窗口、事件计数、唯一会话数、热门条目、上传类型桶、渲染模式计数和截断标记。
- 发布验证应包含构建、代码检查、类型检查，以及至少一次从上传到导出的浏览器冒烟测试。
- 当前仓库的测试先例主要是文档化约束和审计接口手动检查。在把本 PRD 作为更稳定的发布门禁前，应围绕上述接缝补充自动化测试。

## 非目标范围

- 自动主体检测、图像分割或 AI 抠图不属于当前产品形态范围。
- 服务端图片上传、图片持久化、账号系统、作品画廊、协作和项目保存不在范围内。
- 在公开接口中存储或暴露图片内容、原始文件名、IP 地址、原始来源地址或原始审计事件行不在范围内。
- 营销落地页不在范围内；首屏应保持为可用编辑器。
- 批量导出、动画导出、视频导出、SVG 导出和印刷排版流程不在范围内。
- 高级图层管理、可见区域列表、区域命名和批量选区操作不在范围内。
