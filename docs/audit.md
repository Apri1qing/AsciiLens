# AsciiLens Usage Audit

AsciiLens records anonymous product-usage events and exposes a read-only summary API for agents such as OpenClaw.

## What is collected

- Page views
- Image upload and replacement counts
- Image file type and coarse file-size bucket
- Selection creation and deletion counts
- Render mode changes
- PNG export counts
- Browser locale, viewport size, path, referrer host, and user agent

Image contents, file names, IP addresses, and raw referrer URLs are not stored.

## Runtime architecture

- Client events are sent to `POST /api/audit/event`.
- Vercel Serverless Functions write events to Supabase through the service role key.
- Agents fetch metrics through `GET /api/audit/summary`.
- `GET /api/audit/summary` requires `Authorization: Bearer <AUDIT_READ_TOKEN>`.
- If Supabase variables are missing, client analytics returns `204` and the app keeps working.

Daily scheduling lives outside AsciiLens. OpenClaw or another agent can call the summary API on its own cadence, then summarize or route the report in its own workflow.

## Supabase setup

Run this SQL in the Supabase SQL editor:

```sql
create table if not exists public.audit_events (
  id bigint generated by default as identity primary key,
  created_at timestamptz not null default now(),
  event_name text not null check (event_name ~ '^[a-z0-9_.:-]+$'),
  session_id text,
  path text,
  referrer text,
  locale text,
  viewport_width integer,
  viewport_height integer,
  user_agent text,
  metadata jsonb not null default '{}'::jsonb
);

create index if not exists audit_events_created_at_idx
  on public.audit_events (created_at desc);

create index if not exists audit_events_event_name_created_at_idx
  on public.audit_events (event_name, created_at desc);

create index if not exists audit_events_session_created_at_idx
  on public.audit_events (session_id, created_at desc);

alter table public.audit_events enable row level security;
```

Do not add public RLS policies. The browser never talks to Supabase directly; Vercel uses `SUPABASE_SERVICE_ROLE_KEY` on the server.

## Vercel environment variables

Add these variables to the Vercel project:

```text
SUPABASE_URL=https://YOUR_PROJECT.supabase.co
SUPABASE_SERVICE_ROLE_KEY=YOUR_SERVICE_ROLE_KEY
AUDIT_READ_TOKEN=a-long-random-read-token
```

`SUPABASE_SERVICE_ROLE_KEY` must stay server-side only. Do not expose it through Vite environment variables or browser code.

## Summary API

Supported ranges:

- `yesterday` default, previous Asia/Shanghai calendar day
- `today` current Asia/Shanghai calendar day
- `7d` current Asia/Shanghai calendar day plus the previous six days

OpenClaw call example:

```bash
curl "https://asciilens.vercel.app/api/audit/summary?range=yesterday" \
  -H "Authorization: Bearer $AUDIT_READ_TOKEN"
```


## Manual checks

After deploying with environment variables:

```bash
curl -X POST https://asciilens.vercel.app/api/audit/event \
  -H 'Content-Type: application/json' \
  -d '{"eventName":"page_view","sessionId":"manual-check","path":"/","metadata":{"source":"curl"}}'

curl "https://asciilens.vercel.app/api/audit/summary?range=yesterday" \
  -H "Authorization: Bearer $AUDIT_READ_TOKEN"
```

The summary endpoint returns aggregated usage metrics only. It does not expose raw event rows.
